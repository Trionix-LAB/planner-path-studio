Спецификация оборудования (MVP)

Документ описывает варинанты оборудования для работы водолаза.
# 0. Основное окно оборудования
## 0.1 Схемы оборудования
Для каждого оборудования должна быть своя схема UI компонентов, которая должна реализовываться из файла YAML/JSON. Например
```yaml
Zima2R:
  ipAddres:
    inputForm: "text-line"
    inputMask: "0.0.0.0"
    defaultValue: "127.0.0.1"
  inputPort:
    inputForm: "text-line"
    inputMask: "none"
    defaultValue: "28128"
```

Соответственно на основе этого файла выбираются компоненты для отрисовки (текстовое поле `text-line`), значения по умолчанию и маски ввода, если есть. Каждый файл описан в пункте `Схема конфигурации` для каждого оборудования.

## 0.2 Набор оборудования
Должна быть возможность выбрать те или иные виды оборудования. Сохранить выбор и при начале работы в окне с картами передавать информацию о выбранном оборудовании. (напримерв виде JSON-файла)
Для каждого профиля SHOULD поддерживаться несколько инстансов одного типа оборудования (например, два приемника GNSS-UDP с разными портами).
В настройках карты (`Агенты`) источники навигации SHOULD отображать все инстансы активного профиля (для агентов и базовой станции), а не только тип устройства.

## 0.3 Навигация из карты
- На экране карты в окне "Настройки" есть отдельная вкладка "Оборудование".
- На вкладке отображается текущий статус инстансов устройств активного профиля и отдельные кнопки включить/выключить для каждого инстанса.
- С этой вкладки доступна кнопка перехода в основное окно оборудования для детальной настройки профилей и параметров.
- После закрытия основного окна оборудования выполняется возврат в рабочий экран карты (в текущий режим draft/открытая миссия).

# 1. Zima2R
Это система гидроакустической навигации. Имеет две базовые сущности:
- Базовая станция/пеленгационная антенна
- маяк-ответчик
Связь станции с основным приложением происходит с помощью обмена UDP пакетами

## 1.1 Основные настройки
- IP-адрес пеленгационной антенны
- Входной порт команд. Порт для отправки команд из приложения на базовую станцию (если используется)
- Выходной порт данных. Порт для отправки команд от базовой станции в приложение
- Скорость порта ГНСС. Указывает baudrate внешнего ГНСС-компаса (если используется)

## 1.2 Протокол информационного обмена

- **OCON** - Open CONnections. Подключение устройств - открытие соединений. Аналог нажатия кнопки ‘LINK’ в приложении AzimuthSuite. 
Формат команды: `OCON`. Отправлется во входной порт

- **CCON** - Close CONnections. Закрытие всех соединений. Аналог нажатия кнопки ‘LINK’ в приложении AzimuthSuite.
Формат команды: `СCON`. Отправляется во входной порт

- **LHOV** - Location and Heading OVerride.Перегрузка географического положения и азимутального угла. Эта функция может использоваться для задачи вручную географического положения и азимутального угла если пеленгационная антенна статична и не меняет своей ориентации и положения в пространстве.

Формат команды: `LHOV,lat_deg,lon_deg,hdn_deg`. Отправляется во входной порт

| № | Наименование параметра | Описание|
| :---: | :---: | :---: |
| 1 | lat_deg | Географическая широта в градусах в диапазоне от -90 до 90°
| 2 | lon_deg |Географическая долгота в градусах в диапазоне от -180 до 180°
| 3 | hdn_deg |Азимутальный угол в градусах в диапазоне от 0 до 360°

- **AZMLOC** - локальные параметры. Сообщение передается ~1 раз в секунду, когда открыто соединение с пеленгационной антенной или LBL-трансивером.

Формат сообщения: `@AZMLOC,stPressure_mBar,stDepth_m,waterTemp_C,stPitch_deg,stRoll_deg,age,lat_deg,lon_deg,course_deg,speed_mps,age,heading_deg,age,`. Получается из выходного порта

# 3. GNSS-COM
Оборудование для чтения сообщений GNSS-компаса через последовательный порт (COM/TTY) и передачи координат в приложение.

## 3.1 Основные настройки
- Авто COM-порт (`autoDetectPort`): если включен, приложение перебирает доступные последовательные порты и выбирает первый порт с валидным потоком NMEA.
- COM-порт (`comPort`) в ручном режиме задается реальным именем/путем последовательного порта:
  - Windows: `COM3`
  - Linux: `/dev/ttyUSB0`
  - macOS: `/dev/tty.usbserial-XXXX`
- Скорость порта (`baudRate`), по умолчанию `115200`.
- При включенном `autoDetectPort` поле `comPort` блокируется в UI.

## 3.2 Протокол информационного обмена
Для взаимодействия используется стандартный протокол навигационного оборудования NMEA 0183.

Ожидаемое поведение канала:
- чтение входного потока строк NMEA;
- обработка частично пришедших строк (буферизация до завершения строки);
- игнорирование невалидных/неполных строк без остановки всего потока.

## 3.3 Схема конфигурации
| № | Название параметра | Ключ поля | Тип формы для ввода | Маска ввода | Значение по умолчанию |
| :---: | :--- | :--- | :---: | :---: | :---: |
| 1 | Авто COM-порт | `autoDetectPort` | CheckBox | - | `true` |
| 2 | COM-порт | `comPort` | ComboBox | путь/имя порта | `""` |
| 3 | Скорость порта | `baudRate` | Number/TextLine | integer | `115200` |

Примечание: расширенные параметры последовательного порта (`dataBits`, `parity`, `stopBits`) могут быть добавлены позже при необходимости.

# 4. RWLT-COM (uNav RWLT RF Dongle)
Оборудование для чтения сообщений RWLT через последовательный порт (COM/TTY) и передачи телеметрии в приложение.

## 4.1 Основные настройки
- Авто COM-порт (`autoDetectPort`): если включен, приложение пытается выбрать доступный последовательный порт автоматически.
- COM-порт (`comPort`) в ручном режиме:
  - Windows: `COM3`
  - Linux: `/dev/ttyUSB0`
  - macOS: `/dev/tty.usbserial-XXXX`
- Скорость порта (`baudRate`), по умолчанию `38400`.
- Режим (`mode`):
  - `pinger` — агент из `GGA/RMC`;
  - `divers` — водолазы из `PUWV3`.
- Поведение поля `comPort` идентично `GNSS-COM`:
  - при `autoDetectPort=true` поле `comPort` блокируется;
  - при `autoDetectPort=true` значение `comPort` не участвует в выборе порта (игнорируется);
  - при `autoDetectPort=false` поле `comPort` доступно для выбора из списка и ручного ввода;
  - в ручном режиме значение `comPort` обязательно (непустой путь/имя порта).

## 4.2 Контракт информационного обмена (MVP)
- При подключении приложение отправляет `$PUNV0` с `rwlt_mode`:
  - `0` для `pinger`;
  - `1` для `divers`.
- Базовая станция в обоих режимах обновляется из `$PUNV5/$PUWV5`.
- Режим `pinger`:
  - позиция агента из `$GNGGA/$GNRMC`;
  - глубина агента из `GGA.altitude` по правилу `altitude < 0 ? -altitude : 0`.
- Режим `divers`:
  - позиции водолазов из `$PUNV3/$PUWV3`;
  - сопоставление `tID` выполняется по `mission.ui.divers[].beacon_id` (пользовательский диапазон `1..16`, legacy `tID 0..15` нормализуется по правилу `beacon_id = tID + 1` до сопоставления, без эвристик авто-смещения).
- В обоих режимах сообщения `$PRWLA` обновляют позиции буёв RWLT (`buoyId` 1..4); для legacy-потока поддерживается нормализация `0..3 -> 1..4`.
- Буи RWLT отображаются как объекты `rwlt_buoy` (не агенты и не базовая станция) и поддерживают общий object-контракт:
  - доступны в списке `Объекты`;
  - поддерживают `flyTo` / `Переместиться к`;
  - открываются в `Свойства объекта`;
  - поддерживают переименование;
  - поддерживают настройку `marker_size_px`.
- Настройки буёв (`name`, `marker_size_px`) сохраняются в `mission.ui.rwlt_buoys` и восстанавливаются после перезапуска приложения.

## 4.3 Схема конфигурации
| № | Название параметра | Ключ поля | Тип формы | Значение по умолчанию |
| :---: | :--- | :--- | :---: | :---: |
| 1 | Авто COM-порт | `autoDetectPort` | CheckBox | `true` |
| 2 | COM-порт | `comPort` | ComboBox | `""` |
| 3 | Скорость порта | `baudRate` | Number/TextLine | `38400` |
| 4 | Режим работы | `mode` | Select (`pinger`/`divers`) | `pinger` |

## 4.4 Симуляция RWLT
- RWLT-симулятор реализован как отдельный CLI-инструмент over serial (аналогично `gnss-com:sim`).
- Быстрый запуск:
  - `npm run rwlt-com:sim` (Linux/macOS, автоподнятие виртуальной пары через `socat`);
  - `npm run rwlt-com:sim -- --virtual false --port COM12 --baud 38400` (Windows/ручная виртуальная пара).
- Режимы генерации данных:
  - `npm run rwlt-com:sim -- --rwlt-mode pinger` — база (`PUWV5`) + агент (`GGA/RMC`) + буи (`PRWLA`);
  - `npm run rwlt-com:sim -- --rwlt-mode divers --diver-ids 1,2,3` — база (`PUWV5`) + водолазы (`PUWV3`) + буи (`PRWLA`).
- В UI оборудования нужно указать `RWLT-COM -> COM-порт = appPortPath`, который симулятор печатает при старте.
- Детальный контракт CLI/режимов: `docs/features/rwlt-simulator.md`.

# 5. Контракт курса и скорости для HUD/трека

Этот раздел фиксирует единый runtime-контракт формирования `course/speed` для отображения в HUD, ориентации маркера и записи трека.

## 5.1 GNSS-UDP / GNSS-COM

- Курс для агента и базовой станции берется из оборудования по приоритету:
  - при наличии валидного `HDT` (не старше `5` секунд) используется `HDT.headingDeg`;
  - при отсутствии свежего `HDT` курс вычисляется как `Course Over Ground (COG)` по последовательным координатам и времени.
- Таймаут свежести `HDT` фиксирован и равен `5` секундам.
- Скорость для GNSS-источников вычисляется как `Speed Over Ground (SOG)` по последовательным координатам и времени.
- При первой валидной точке (или при невалидном интервале времени между точками) используются `speed=0`, `course=0` до появления валидной пары точек.

## 5.2 Zima2R

- Базовая станция (`AZMLOC`):
  - курс берется из `AZMLOC.course_deg`;
  - скорость берется из `AZMLOC.speed_mps`.
- Агент (`AZMREM`):
  - курс и скорость вычисляются как `COG/SOG` по последовательным координатам `AZMREM` для конкретного `rem_addr`.
  - курс/скорость из `AZMLOC` не подмешиваются в телеметрию агента.

## 5.3 RWLT-COM

- Базовая станция (оба режима):
  - курс берется из `PUWV5.gnssCrs`;
  - скорость берется из `PUWV5.gnssSog` (км/ч -> м/с).
- Режим `pinger` (агент из `GGA/RMC`):
  - `Course Overground`: по COG из последовательных координат агента;
  - `Speed Overground`: по SOG из последовательных координат агента.
- Режим `divers` (агенты из `PUWV3`):
  - `Course Overground`: из `PUWV3.tCrs`, если он валиден; иначе по COG из последовательных координат конкретного `tID`;
  - `Speed Overground`: по SOG из последовательных координат конкретного `tID`.
