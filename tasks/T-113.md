# T-113: Async TIFF decoder in Electron main process

**Issue:** local-fix (no GitHub issue)
**Spec:** no behavior-contract changes (runtime/perf fix)

## Why
- Synchronous `decodeTiffToPng` in Electron main process can block event loop on large TIFF files.
- `tools/test_tifs/002.tif` exposed decoder-path instability across fallback strategies.

## Goal
- Keep TIFF decoding reliable in packaged desktop builds without requiring users to install external CLI tools.
- Prevent render corruption for very large decoded rasters in Chromium-based renderer.

## Plan
- [x] Add async wrapper `decodeTiffToPngAsync` in `electron/tiff-decoder.cjs` (worker thread).
- [x] Use async decoder in IPC handler.
- [x] Add adaptive timeout for very large TIFF files in built-in decoder.
- [x] Return `sharp` to priority decoder path for TIFF-to-PNG conversion.
- [x] Configure `sharp` for large images (`unlimited: true`, `limitInputPixels: false`).
- [x] Remove external CLI fallback (`magick`/`convert`/`sips`) to avoid external runtime dependencies.
- [x] Add `asarUnpack` for `sharp` and `@img` in electron-builder config.
- [x] Add resize guard in sharp path to fit renderer-safe limits for huge rasters.

## Acceptance
- [x] `planner:raster:convertTiffBase64ToPngBase64` works without requiring external tools.
- [x] `tools/test_tifs/002.tif` decodes through sharp path.
- [x] Resulting PNG is resized to renderer-safe dimensions for huge inputs.
- [x] Built app does not depend on ImageMagick/sips presence for TIFF decoding.
