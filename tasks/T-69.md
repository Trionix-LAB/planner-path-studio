# T-69: Track rendering optimization (R-040)

**Issue:** [#69](https://github.com/Trionix-LAB/planner-path-studio/issues/69)
**Branch:** `chore/map-69-track-rendering-optimization`
**Spec:** R-040

---

## Why

Трек рендерится через набор `<Polyline>` в `src/components/map/MapCanvas.tsx` — один React-Leaflet компонент на каждый сегмент. При 200 000 точек SVG-рендеринг (даже с `preferCanvas`) деградирует: каждый pan/zoom пересчитывает все точки через Leaflet-проекцию, React-reconciler сравнивает массивы, а браузер перерисовывает длинную ломаную. Результат — лаги pan/zoom, нарушение R-040.

**Текущее состояние (из аудита A-0067):**
- `preferCanvas={true}` уже выставлен — хорошо, canvas-рендер активен.
- Отсутствует какая-либо decimation/simplification.
- Отсутствует throttle обновлений трека.
- `buildTrackSegments` не фильтрует точки по вьюпорту и зуму.

## Problem analysis

### Где проблема

**1. `src/features/mission/model/adapters.ts`, `buildTrackSegments` (строки 247–273)**

`useMemo` пересчитывается при **каждом новом fix** (т.к. `trackPointsByTrackId` меняется). Результат — все 200k точек прогоняются через mapper на каждой GPS-точке. Нет throttle.

**3. `src/components/map/MapCanvas.tsx`, строки 1534–1545**

```tsx
{layers.track &&
  trackSegments.map((segment, index) => (
    <Polyline
      key={`track-segment-${segment.trackId}-${index}`}
      positions={segment.points}   // ← весь массив — реактивный prop
      pathOptions={{ color: segment.color, weight: styles.track.width_px }}
    />
  ))}
```

Каждый пересчёт `trackSegments` триггерит React-reconciler + Leaflet перепроекцию всех точек.

**Нет decimation — нет zoom-aware simplification.**

## Goal

Обеспечить отзывчивый pan/zoom при треке ~200 000 точек в рамках R-040.

---

## Solution strategy

Реализуем **zoom-aware decimation** (упрощение геометрии по зуму) + **throttle обновления** трека в UI. Это минимально-инвазивный подход, не требующий замены Leaflet или перехода на WebGL.

### Алгоритм decimation: Ramer–Douglas–Peucker (RDP)

- Стандартный алгоритм упрощения ломаной с контролируемым порогом `epsilon`.
- `epsilon` = метры на пиксель при текущем зуме → чем больше зум, тем меньше epsilon, тем больше точек сохраняется.
- Реализовать в **географических координатах** (lat/lon) с haversine-расстоянием до отрезка.
- Применять **один раз** при изменении зума или массива точек — результат кешировать через `useMemo`.

### Throttle обновления трека

- `trackSegments` пересчитывается при каждом новом fix (1–10 Hz). Для UI достаточно 1 раз в 500–1000 мс.
- Реализовать через кастомный хук `useThrottledValue` или встроить throttle в `useMemo` через `useRef`-таймер.
- Активный трек: throttle 500 мс. Завершённые треки: без throttle (точки не меняются).

### Архитектура изменений

```
trackPointsByTrackId (сырые данные, обновляется каждый fix)
        │
        ▼ useThrottledValue(500ms) — для активных треков
throttledTrackPoints
        │
        ▼ buildTrackSegments (разбивка по segment_id)
rawSegments
        │
        ▼ decimateSegments(zoom, metersPerPixel) — RDP
decimatedSegments
        │
        ▼ filterVisibleTrackSegments (hideTrackIds)
visibleTrackSegments → <Polyline positions={...} />
```

## Plan

### Шаг 1: Реализовать RDP decimation в `src/features/mission/model/trackDecimation.ts` (НОВЫЙ файл)

```ts
import type { TrackSegment } from './adapters';

/**
 * Ramer–Douglas–Peucker simplification для массива [lat, lon] точек.
 * epsilon — порог в градусах (lat/lon пространство, аппроксимация).
 * Используем перпендикулярное расстояние в декартовом приближении (достаточно для небольших областей).
 */
export function rdpSimplify(
  points: Array<[number, number]>,
  epsilon: number,
): Array<[number, number]> {
  if (points.length <= 2) return points;
  // ... рекурсивный RDP
}

/**
 * Применить decimation ко всем сегментам.
 * epsilonDeg — порог в градусах; вычисляется из metersPerPixel через approximateDegreesPerMeter.
 */
export function decimateSegments(
  segments: TrackSegment[],
  epsilonDeg: number,
): TrackSegment[] {
  if (epsilonDeg <= 0) return segments;
  return segments.map((seg) => ({
    ...seg,
    points: rdpSimplify(seg.points, epsilonDeg),
  }));
}

/**
 * Вычислить epsilon в градусах из metersPerPixel карты.
 * Один пиксель = metersPerPixel метров. Хотим прореживать точки, которые ближе 1px друг к другу.
 * 1 градус широты ≈ 111 320 м.
 */
export function epsilonDegFromMetersPerPixel(metersPerPixel: number): number {
  return metersPerPixel / 111_320;
}
```

**Детали реализации RDP:**
- Перпендикулярное расстояние от точки до отрезка: использовать декартово приближение (умножать на `cos(lat_rad)` для долготы). Это точно для областей < 100 км.
- При `points.length <= 2` — возвращать как есть.
- Рекурсия: найти точку с максимальным перпендикулярным расстоянием, если > epsilon — рекурсивно обработать обе части, иначе вернуть только endpoints.
- Итеративная версия на стеке (не рекурсивная) — предпочтительна для длинных треков (избежать stack overflow на 200k точек).

**Важно:** decimation работает в lat/lon-пространстве с декартовым приближением — это корректно для расстояний до нескольких сотен км. Haversine не нужен.

### Шаг 2: Реализовать `useThrottledValue` в `src/hooks/useThrottledValue.ts` (НОВЫЙ файл)

```ts
import { useEffect, useRef, useState } from 'react';

/**
 * Возвращает throttled-копию значения.
 * Обновляется немедленно при первом рендере, затем не чаще чем intervalMs.
 */
export function useThrottledValue<T>(value: T, intervalMs: number): T {
  const [throttled, setThrottled] = useState<T>(value);
  const lastUpdateRef = useRef<number>(0);
  const pendingRef = useRef<ReturnType<typeof setTimeout> | null>(null);

  useEffect(() => {
    const now = Date.now();
    const sinceLastUpdate = now - lastUpdateRef.current;

    if (sinceLastUpdate >= intervalMs) {
      lastUpdateRef.current = now;
      setThrottled(value);
    } else {
      if (pendingRef.current) clearTimeout(pendingRef.current);
      pendingRef.current = setTimeout(() => {
        lastUpdateRef.current = Date.now();
        setThrottled(value);
        pendingRef.current = null;
      }, intervalMs - sinceLastUpdate);
    }

    return () => {
      if (pendingRef.current) clearTimeout(pendingRef.current);
    };
  }, [value, intervalMs]);

  return throttled;
}
```

**Требования:**
- Первое значение применяется мгновенно (нет задержки при монтировании).
- При остановке трека (точки перестают поступать) — финальное значение всё равно должно применяться (через timeout). Это обеспечивает отображение последней точки трека.
- Throttle должен быть стабильным (не сбрасываться при каждом рендере).

### Шаг 3: Интегрировать `useMapZoom` в `src/components/map/MapCanvas.tsx`

Внутри `MapCanvas` уже есть `useMap()` в дочерних компонентах. Нужно получать текущий зум для вычисления epsilon.

Добавить хук внутрь `MapCanvasInner` (или как называется внутренний компонент):

```tsx
// Получить metersPerPixel из текущего зума карты
const map = useMap();
const [metersPerPixel, setMetersPerPixel] = useState<number>(
  computeMetersPerPixelFromMap(map) // уже есть в scaleUtils.ts
);

useEffect(() => {
  const update = () => setMetersPerPixel(computeMetersPerPixelFromMap(map));
  map.on('zoomend', update);
  map.on('moveend', update);
  return () => {
    map.off('zoomend', update);
    map.off('moveend', update);
  };
}, [map]);
```

Затем применять decimation перед рендером трека:

```tsx
const decimatedTrackSegments = useMemo(() => {
  const epsilon = epsilonDegFromMetersPerPixel(metersPerPixel);
  return decimateSegments(trackSegments, epsilon);
}, [trackSegments, metersPerPixel]);

// В JSX:
{layers.track &&
  decimatedTrackSegments.map((segment, index) => (
    <Polyline key={...} positions={segment.points} pathOptions={...} />
  ))}
```

**Внимание:** `computeMetersPerPixelFromMap` уже существует в `src/components/map/scaleUtils.ts` — переиспользовать, не дублировать.

## Guardrails

- **Не менять `buildTrackSegments`** — только добавить шаг decimation после него.
- **Не менять `trackRecorder.ts`** — это отдельная оптимизация (O(n) spread, другой P0 из аудита).
- **Не менять `filterVisibleTrackSegments`** — decimation должен происходить ДО фильтрации видимости, чтобы скрытые треки не тратили ресурсы на decimation. (Или ПОСЛЕ — это приемлемо тоже, т.к. decimation дешевле рендера.)
- **`trackPointsByTrackId` без throttle** должен использоваться везде, кроме рендера: autosave, export, trackWriter (T-68), статистика.
- **Не применять throttle к завершённым трекам** — их точки не меняются, `useThrottledValue` для них нейтрален (сразу возвращает значение, т.к. обновлений нет).
- **Итеративный RDP** (не рекурсивный) — обязательно. Рекурсия на 200k точек при малом epsilon вызовет stack overflow.
- **`computeMetersPerPixelFromMap`** уже есть в `src/components/map/scaleUtils.ts` — не дублировать.
- **`preferCanvas={true}`** уже выставлен в `MapContainer` — не трогать.
- Web-платформа: decimation работает одинаково в web и Electron — нет платформенных различий.
- Не менять формат данных `TrackSegment` — только прореживать `points` внутри.

## Notes

- RDP — хорошо изученный алгоритм с `O(n log n)` в среднем случае и `O(n²)` в худшем (колинеарные точки). Для треков с реальным GPS-шумом всегда `O(n log n)`.
- Альтернативы (Visvalingam–Whyatt, радиальное прореживание, WebGL) рассматривались — RDP оптимален для данного случая: прост в реализации, хорошо тестируется, достаточен для 200k точек.
- Если после реализации bench покажет недостаточность — можно добавить viewport culling (отсечение точек вне текущего bbox карты) как отдельный шаг.
- Context: `docs/audits/A-0067-arch-performance.md` (P0 Track rendering), issue #67.
