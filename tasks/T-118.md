# T-118: Equipment data logging per mission

**Issue:** [#118](https://github.com/Trionix-LAB/planner-path-studio/issues/118)
**Branch:** `feat/mission-devices-118-equipment-logging`
**Spec:** `R-069` (добавить в `spec/spec.md`)

---

## Why

Сейчас все сырые пакеты от оборудования (AZMLOC/AZMREM/NMEA-sentences) обрабатываются в памяти и теряются — в файлах миссии сохраняются только разобранные GPS-точки в CSV-треках. Отладка нештатных ситуаций, послеполётный анализ и верификация данных невозможны без сырых логов устройств.

---

## Goal

Для каждого `DeviceInstance` активного `EquipmentProfile` вести отдельный `.log`-файл в папке миссии. Логирование запускается вместе со стартом записи трека и останавливается при его остановке. Файлы доступны для скачивания из UI.

---

## Architecture

### Куда добавить код

```
src/features/mission/model/
  equipmentLogger.ts          ← НОВЫЙ: фабрика EquipmentLogger, буфер, flush
  telemetry.ts                ← ИЗМЕНИТЬ: добавить onRawPacket() в TelemetryProvider
src/features/devices/model/
  types.ts                    ← ИЗМЕНИТЬ: добавить logging_enabled в EquipmentProfile
  settings.ts                 ← ИЗМЕНИТЬ: дефолт logging_enabled=true, миграция V3→V3
```

Хуки / компоненты, которые управляют lifecycle записи, потребуют дополнительной интеграции — найти через `useMissionAutosave` и места вызова `trackRecorderReduce` с `type: 'start'/'stop'/'pause'/'resume'`.

### Путь хранения

```
<mission-rootPath>/logs/equipment/<device-instance-id>.log
```

Аналогично CSV-трекам: `<rootPath>/tracks/<agent_uid>-track-NNNN.csv`.
Ключ `platform.fileStore` → `<rootPath>/logs/equipment/<device-instance-id>.log`.

### Формат каждой строки

```
2026-03-06T14:23:01.123Z {"schema_id":"zima2r","raw":"$AZMLOC,0,59.93,30.33,..."}
```

- Timestamp: `new Date().toISOString()` (UTC, ISO 8601 с миллисекундами).
- JSON-объект: `{ schema_id: string; raw: string }` — минимальный контракт. Дополнительные поля допустимы, но не обязательны.
- Строки разделяются `\n`.
- Lazy-create: файл создаётся при первой записи, не при открытии миссии.

    export type RawTelemetryPacket = {
      schema_id: string;   // 'zima2r' | 'gnss-udp' | 'gnss-com'

  - **Важно:** `emitRawPacket` вызывается ДО `if (!enabled || simulateConnectionError) return` — чтобы логировались все пакеты, включая отладочные.
    Нет — пересмотреть: логировать только когда `enabled === true`, чтобы не засорять лог во время симуляции ошибок.

- [ ] **Добавить `logging_enabled` в `EquipmentProfile`** (`src/features/devices/model/types.ts`):
  ```ts

    id: string;
    name: string;
    device_instance_ids: string[];
    logging_enabled?: boolean;  // default: true (undefined === true)
  };
  ```
  В `settings.ts` при чтении профиля нормализовать: `profile.logging_enabled ?? true`.
  Поскольку поле опциональное, существующие сохранённые профили без поля автоматически воспринимаются как `true` — отдельная миграция не нужна.

- [ ] **Создать `src/features/mission/model/equipmentLogger.ts`**:

  ```ts
  import type { FileStoreBridge } from '@/platform/contracts';

  export type EquipmentLoggerOptions = {
    rootPath: string;
    deviceInstanceId: string;
    schemaId: string;
    fileStore: FileStoreBridge;
  };

  export type EquipmentLogger = {
    /** Вызывается при получении сырого пакета (буферизует, не блокирует). */
    write: (raw: string) => void;
    /** Принудительный сброс буфера на диск. */
    flush: () => Promise<void>;
    /** Сброс буфера + остановка авто-flush. */
    stop: () => Promise<void>;
  };

  export const createEquipmentLogger = (options: EquipmentLoggerOptions): EquipmentLogger => { ... };
  ```

  **Детали реализации `createEquipmentLogger`:**
  - Буфер: `string[]`, накапливает строки вида `<ISO-timestamp> <JSON>\n`.
  - `write(raw)`: добавляет строку в буфер. Если буфер достиг 50 строк — асинхронно вызывает внутренний `_flush()` (не блокирует caller).
  - Авто-flush: `setInterval(_flush, 1000)` запускается при первом `write()` (lazy). Хранится `intervalId`.
  - `_flush()` (приватный):
    1. Если буфер пуст — выходит.
    2. Извлекает все строки (`const lines = buffer.splice(0)`).
    3. Вызывает `fileStore.appendText(logPath, lines.join(''))`.
    4. Если `appendText` бросает — строки возвращаются в начало буфера (`buffer.unshift(...lines)`) для повторной попытки.
  - `flush()` (публичный): вызывает `_flush()`, возвращает Promise.
  - `stop()`: очищает `setInterval`, вызывает `_flush()`, возвращает Promise.
  - `logPath`: `${options.rootPath}/logs/equipment/${options.deviceInstanceId}.log`.
  - **Не бросать** из `write()` — все ошибки проглатываются тихо (console.warn допустим).

- [ ] **Интегрировать logger в lifecycle трека** — найти место, где вызываются события `start`/`stop`/`pause`/`resume` трека, и там же создавать/останавливать `EquipmentLogger` для каждого DeviceInstance активного профиля.

  Точки интеграции (исследовать в ходе реализации):
  - `src/features/mission/hooks/useMissionAutosave.ts` или компонент, вызывающий `trackRecorderReduce({ type: 'start' })`.
  - Хук или эффект должен:
    1. При `start`/`resume` для агента — создавать (или переиспользовать) `EquipmentLogger` для каждого `DeviceInstance` активного профиля, у которого `logging_enabled !== false`.
    2. Подписываться через `telemetryProvider.onRawPacket(packet => logger.write(packet.raw))`.
    3. При `pause`/`stop` — вызывать `logger.flush()` + отписываться.
    4. При `stopAll` — flush всех активных логгеров.
    5. При размонтировании — `logger.stop()` для всех.

  **Если в профиле несколько устройств одного schema_id** — каждый `DeviceInstance` имеет уникальный `id`, логгер создаётся на `instance.id`, но `schema_id` нужен для метаданных в `raw`-строке. При подписке на `onRawPacket` нужно фильтровать пакеты по `packet.schema_id === instance.schema_id` — это единственный способ сопоставить пакет с экземпляром, если у одного провайдера один `schema_id`. Если у провайдера нет способа различить экземпляры (а его нет — один провайдер = один schema_id), то **все экземпляры одного schema_id пишут в один файл** (первый по порядку в `device_instance_ids`). Это ограничение MVP, зафиксировать в комментарии.

## Acceptance Criteria (из Issue #118)

- [ ] Для миссии с профилем из 2 устройств создаются 2 файла `logs/equipment/<id>.log`.
- [ ] Каждая строка начинается с ISO 8601 UTC timestamp.
- [ ] JSON-payload содержит `schema_id` и `raw`.
- [ ] Файлы можно скачать из интерфейса миссии.
- [ ] При `logging_enabled: false` файлы не создаются.
- [ ] Логирование останавливается при `stop`/`pause` и возобновляется при `resume`/`start`.
- [ ] `npm run build` и `npm run lint` без новых ошибок.
- [ ] На web-платформе данные сохраняются в localStorage через `platform.fileStore`.

## Guardrails

- Не использовать `electron-log` — он для app-level debug-логов (ADR-0001/R-000). Equipment-логи — данные миссии.
- Не писать напрямую в `fs` или IPC — только через `platform.fileStore`.
- Не создавать новые экраны — интегрировать в существующий UI.
- Не изменять CSV-формат треков.
- Буферизация только на стороне рендерера; IPC не вызывать чаще чем раз в секунду на устройство.
