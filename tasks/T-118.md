# T-118: Equipment data logging per mission

**Issue:** [#118](https://github.com/Trionix-LAB/planner-path-studio/issues/118)
**Branch:** `feat/mission-devices-118-equipment-logging`
**Spec:** `R-069` (добавить в `spec/spec.md`)

---

## Why

Сейчас все сырые пакеты от оборудования (AZMLOC/AZMREM/NMEA-sentences) обрабатываются в памяти и теряются — в файлах миссии сохраняются только разобранные GPS-точки в CSV-треках. Отладка нештатных ситуаций, послеполётный анализ и верификация данных невозможны без сырых логов устройств.

---

## Goal

Для каждого `DeviceInstance` активного `EquipmentProfile` вести отдельный `.log`-файл в папке миссии. Логирование работает always-on (не зависит от статуса записи трека и не имеет отдельного переключателя отключения). Файлы доступны для скачивания из UI.

---

## Architecture

### Куда добавить код

```
src/features/mission/model/
  equipmentLogger.ts          ← НОВЫЙ: фабрика EquipmentLogger, буфер, flush
  telemetry.ts                ← ИЗМЕНИТЬ: добавить onRawPacket() в TelemetryProvider
src/features/devices/model/
  types.ts                    ← ИЗМЕНИТЬ: EquipmentProfile без отдельного флага отключения логов
  settings.ts                 ← ИЗМЕНИТЬ: убрать нормализацию logging_enabled
```

Хуки / компоненты, которые управляют lifecycle миссии, потребуют дополнительной интеграции — найти точки открытия/закрытия миссии и смены активного профиля оборудования.

### Путь хранения

```
<mission-rootPath>/logs/equipment/<device-instance-id>.log
```

Аналогично CSV-трекам: `<rootPath>/tracks/<agent_uid>-track-NNNN.csv`.
Ключ `platform.fileStore` → `<rootPath>/logs/equipment/<device-instance-id>.log`.

### Формат каждой строки

```
2026-03-06T14:23:01.123Z {"profile_name":"Workboat A","raw":"$AZMLOC,0,59.93,30.33,..."}
```

- Timestamp: `new Date().toISOString()` (UTC, ISO 8601 с миллисекундами).
- JSON-объект: `{ profile_name: string; raw: string }` — минимальный контракт. Дополнительные поля допустимы, но не обязательны.
- Строки разделяются `\n`.
- Lazy-create: файл создаётся при первой записи, не при открытии миссии.

- [x] **Создать `src/features/mission/model/equipmentLogger.ts`**:

  ```ts
  import type { FileStoreBridge } from '@/platform/contracts';

  export type EquipmentLoggerOptions = {
    rootPath: string;
    deviceInstanceId: string;
    profileName: string;
    fileStore: FileStoreBridge;
  };

  export type EquipmentLogger = {
    /** Вызывается при получении сырого пакета (буферизует, не блокирует). */
    write: (raw: string) => void;
    /** Принудительный сброс буфера на диск. */
    flush: () => Promise<void>;
    /** Сброс буфера + остановка авто-flush. */
    stop: () => Promise<void>;
  };

  export const createEquipmentLogger = (options: EquipmentLoggerOptions): EquipmentLogger => { ... };
  ```

  **Детали реализации `createEquipmentLogger`:**
  - Буфер: `string[]`, накапливает строки вида `<ISO-timestamp> <JSON>\n`.
  - `write(raw)`: добавляет строку в буфер. Если буфер достиг 50 строк — асинхронно вызывает внутренний `_flush()` (не блокирует caller).
  - Авто-flush: `setInterval(_flush, 1000)` запускается при первом `write()` (lazy). Хранится `intervalId`.
  - `_flush()` (приватный):
    1. Если буфер пуст — выходит.
    2. Извлекает все строки (`const lines = buffer.splice(0)`).
    3. Вызывает `fileStore.appendText(logPath, lines.join(''))`.
    4. Если `appendText` бросает — строки возвращаются в начало буфера (`buffer.unshift(...lines)`) для повторной попытки.
  - `flush()` (публичный): вызывает `_flush()`, возвращает Promise.
  - `stop()`: очищает `setInterval`, вызывает `_flush()`, возвращает Promise.
  - `logPath`: `${options.rootPath}/logs/equipment/${options.deviceInstanceId}.log`.
  - **Не бросать** из `write()` — все ошибки проглатываются тихо (console.warn допустим).

## Guardrails

- Не использовать `electron-log` — он для app-level debug-логов (ADR-0001/R-000). Equipment-логи — данные миссии.
- Не писать напрямую в `fs` или IPC — только через `platform.fileStore`.
- Не создавать новые экраны — интегрировать в существующий UI.
- Не изменять CSV-формат треков.
- Буферизация только на стороне рендерера; IPC не вызывать чаще чем раз в секунду на устройство.
