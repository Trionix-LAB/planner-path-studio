# T-108: Coordinate input in multiple CRS (WGS84 / SK-42 / GSK-2011)

**Issue:** #108
**Spec:** New requirement

## Context
* Internal storage and most UI currently use WGS84 lat/lon (`EPSG:4326`) as the source of truth.
  - Confirmed by docs and existing code paths (mission format + MapObject geometry).
* Geometry editing by coordinates already exists for routes/zones as tables in `src/components/map/MapObjectProperties.tsx`.
  - Today these dialogs accept only WGS84 lat/lon and validate ranges `lat [-90..90]`, `lon [-180..180]`.
* Marker coordinates are created/edited mostly via map interactions; there is no CRS-aware coordinate editor.

Issue goal is to allow coordinate entry in multiple coordinate reference systems and auto-convert to the internal WGS84 representation.

## Goal
* Add a CRS selector for coordinate entry for:
  - markers
  - route point tables
  - zone vertex tables
* Add CRS + coordinate format selector in read-only lane vertices viewer (`Вершины галсов`) for display conversion.
* Add coordinate format selector (radio buttons) for input/display in coordinate editors:
  - `Десятичные градусы`
  - `Градусы и десятичные минуты`
  - `Градусы, минуты и секунды`
* Add input masks for coordinate fields according to selected format (for lat/lon).
* On submit, convert entered coordinates from selected CRS -> WGS84 and save into object geometry.
* Show both:
  - the user-entered coordinate values in the selected CRS
  - the converted WGS84 values (read-only preview)
* Add conversion tests with control points for each supported CRS.

## MVP decision (recommended)
* Internal CRS stays WGS84 (`EPSG:4326`) for all persisted geometry.
* We do NOT store "original CRS coordinates" per point in mission format for MVP.
  - Instead, we store geometry in WGS84 and compute the "selected CRS view" on the fly.
  - Persist only the user's last-selected CRS in app settings (so the UI reopens in that CRS).
* CRS support scope:
  - `WGS84 (EPSG:4326)`
  - `СК-42 (Pulkovo 1942)`
  - `ГСК-2011`

## Open questions (must be resolved in implementation)
1) Input format for SK-42 / GSK-2011:
   - Geographic lat/lon degrees in those datums? (recommended for MVP)
   - Or projected coordinates (Gauss-Kruger / UTM-like)? (likely out of scope)
2) Exact EPSG codes + transform parameters:
   - SK-42 commonly maps to `EPSG:4284` (Pulkovo 1942) for geographic.
   - GSK-2011 EPSG code must be confirmed (likely geographic; depends on EPSG registry).
   - Datum shift accuracy varies; we must document expected error and provide a disclaimer.

## Resolution (implemented)
* Для MVP принят **географический ввод `lat/lon`** для `СК-42` и `ГСК-2011`; projected-варианты (Gauss-Kruger/UTM-like) оставлены вне scope.
* Введен модуль `src/features/geo/crs.ts` c pure-конвертацией:
  - `WGS84 <-> СК-42`: глобально калиброванная совместимая модель (`lat/lon` delta) с обратимым преобразованием;
  - `WGS84 <-> ГСК-2011`: географическая конвертация с нулевой геоцентрической трансляцией и параметрами эллипсоида GSK-2011 (EPSG:1025).
* Ограничение точности и применимость параметров зафиксированы в `spec/spec.md` (R-060).
* Для устойчивости на глобальном наборе контрольных точек (`tools/coordinates.md`) СК-42 использует единый совместимый калиброванный режим без регионального переключения модели.

## Technical approach
* Use a JS projection library (`proj4`) to convert between CRSs.
  - Add only the minimum definitions needed for the supported CRSs.
  - If a CRS is not supported or transform is unavailable, block submit with a clear error.
* Create a small conversion module:
  - `src/features/geo/crs.ts`:
    - `type CrsId = 'wgs84' | 'sk42' | 'gsk2011'`
    - `getCrsLabel(crs)`
    - `convertPoint({ lat, lon }, from, to) -> { lat, lon }`
    - `formatPoint(crs, point, precision)`
  - Keep it pure and unit-testable.

## UI design (recommended)
The CRS-aware coordinate input must be exposed as a dedicated "Build by coordinates" menu for each entity.

Entry point (explicit requirement):
* For each entity (marker/route/zone) this coordinate input window is opened via **right-click (context menu)** on the entity name in the **TopBar**.

In the opened window (explicit requirements):
* User can:
  - choose the source coordinate system (ComboBox)
  - input point(s) as numeric fields with range validation
  - add new points and delete existing points independently (per-row add/remove)
  - click action button `Ок`, which only closes the coordinate window

Behavior:
* On `Сохранить изменения` in the right panel, ALL entered points are converted from the selected source CRS to the map target CRS **WGS84 (EPSG:4326)** and applied to the entity geometry.
* Changing only CRS and/or coordinate format in the coordinate window updates values/preview in real time and does not apply geometry automatically.

Recommended presentation:
* Marker: one-point form (lat/lon).
* Route: points table (>=2 rows).
* Zone: vertices table (>=3 rows).
* Zone lane vertices: read-only table with switchable CRS/format for viewing.
* Optionally show a read-only preview of the computed WGS84 values before applying.

Persist UI preference:
* Store last-selected CRS in app settings (e.g. `AppSettingsV1.defaults.coordinates.input_crs`).
* Store default coordinate format in app settings and reuse it as initial value in editors.
