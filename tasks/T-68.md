# T-68: Incremental track persistence

**Issue:** [#68](https://github.com/Trionix-LAB/planner-path-studio/issues/68)
**Branch:** `chore/mission-68-incremental-track-persistence`
**Spec:** R-036, R-040 (обновить формулировки)

---

## Why

`writeCheckpoint` в `MissionRepository` на каждом autosave целиком перезаписывает CSV каждого трека через `store.writeText`. При треке на 200 000 точек это означает сериализацию и запись ~10–20 МБ на каждый autosave-цикл (~каждые 1–2 с). Результат — IO-фризы, рост GC-давления, риск несоответствия R-040.

Решение: активные треки пишутся инкрементально (append по мере поступления точек), checkpoint перезаписывает полностью только завершённые треки; для активных треков checkpoint синхронизирует только `mission.json` + GeoJSON.

---

## Problem analysis

### Где проблема

`src/features/mission/model/repository.ts`, функция `writeCheckpoint` (line 305–333):

```ts
for (const track of mission.tracks) {
  const points = bundle.trackPointsByTrackId[track.id] ?? [];
  const csvPath = joinPath(rootPath, track.file);
  await store.writeText(csvPath, toCsvTrack(points));  // ← полная перезапись
}
```

Вызывается каждый autosave-цикл через `saveMission` → `saveMissionInternal` → `writeCheckpoint`.

Дополнительный риск: точки активных треков хранятся в React-state через O(n) spread в `trackRecorder.ts` — это отдельная проблема (P0 из audit), но она вне scope данного issue.

### Целевая архитектура

Разделить persistence треков на два уровня:

1. **Track append writer** — для каждого активного трека отдельный объект, который буферизует входящие точки и периодически пишет их в CSV через `appendText`.
2. **Checkpoint** — для активных треков пропускает перезапись CSV; для завершённых треков (`ended_at !== null`) перезаписывает целиком (они больше не меняются).

При открытии миссии: читать CSV целиком (без изменений) — файл уже содержит все сохранённые точки.

- [x] **Создать `src/features/mission/model/trackWriter.ts`** — инкрементальный писатель треков:

  ```ts
  import type { FileStoreBridge } from '@/platform/contracts';
  import { type TrackPoint } from './types';

  const CSV_HEADER = 'timestamp,lat,lon,segment_id,depth_m,sog_mps,cog_deg';

  export type TrackWriterOptions = {
    trackPath: string;       // абсолютный путь в fileStore
    fileStore: FileStoreBridge;
    flushIntervalMs?: number;  // default: 1000
    maxBufferLines?: number;   // default: 50
  };

  export type TrackWriter = {
    /** Записать новые точки (после последнего вызова). Вызывать при каждом fixReceived. */
    append: (newPoints: TrackPoint[]) => void;
    /** Принудительный сброс буфера. */
    flush: () => Promise<void>;
    /** Сброс + остановка авто-flush. */
    stop: () => Promise<void>;
    /** Полная перезапись файла (вызывать при stop/pause трека). */
    rewrite: (allPoints: TrackPoint[]) => Promise<void>;
  };

  export const createTrackWriter = (options: TrackWriterOptions): TrackWriter => { ... };
  ```

  **Детали реализации:**

  - При первом `append()` — если файл не существует (lazy), записать заголовок CSV через `appendText(path, CSV_HEADER + '\n')`.
  - Буфер: строки CSV. `append(newPoints)` форматирует точки в CSV-строки и добавляет в буфер.
  - `_flush()`: если буфер пуст — noop. Иначе — `appendText(path, lines.join('\n') + '\n')`. При ошибке — строки возвращаются в буфер (retry).
  - Авто-flush: `setInterval(_flush, flushIntervalMs)`, запускается lazy при первом `append`.
  - При буфере ≥ `maxBufferLines` — немедленный async flush.
  - `stop()`: clearInterval + flush.
  - `rewrite(allPoints)`: остановить авто-flush, `writeText(path, toCsvTrack(allPoints))`, сбросить буфер.
  - `append()` не бросает — ошибки только через `console.warn`.

  Вспомогательная функция `toCsvLine(point: TrackPoint): string` — форматирует одну точку (без заголовка). Вынести из `repository.ts` или продублировать локально. Позже можно переиспользовать из общего модуля.

## Guardrails

- Не менять формат CSV на диске — заголовок и строки должны соответствовать `toCsvTrack` из `repository.ts`.
- Не менять формат WAL (`logs/wal/current.wal`) — он по-прежнему содержит полный snapshot треков в памяти (для recovery). WAL и CSV-файлы — независимые механизмы.
- При открытии миссии (`openMission`) CSV по-прежнему читается целиком через `parseCsvTrack` — изменений нет.
- `appendTrackPoints` передаёт только **новые** точки (не весь массив из state) — иначе теряется смысл инкрементальности.
- `finalizeTrack`/`finalizeAllTracks` должны вызываться **до** `saveMission` при stop/close, чтобы финальный checkpoint видел уже записанный файл (хотя checkpoint для активных треков и пропускается, финализация нужна для консистентности на диске).
- Web-платформа: `appendText` в `localStorage` реализован — проверить что поведение корректно (конкатенация строк к ключу). Если нет — зафиксировать известное ограничение в комментарии, не фиксить в этом PR.
- Не трогать `trackRecorder.ts` (O(n) spread) — это отдельная оптимизация (другой P0 из audit).
